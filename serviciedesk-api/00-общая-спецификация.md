# Общая спецификация API IntraService

## Введение

Этот документ описывает общие принципы работы с REST API системы IntraService.

## Базовый URL

Все запросы к API отправляются по следующему адресу:

```
https://{domain}.intraservice.ru/api/
```

Где `{domain}` — адрес вашей установки IntraService (например, `mycompany`).

**Примеры:**
- Продакшен: `https://mycompany.intraservice.ru/api/task`
- Локальная разработка: `http://localhost/IntraService4.40p2/api/task`

## Авторизация

Для доступа к API используется базовая HTTP-авторизация (Basic Auth).

### Подробнее о базовой авторизации

Запросы должны содержать заголовок `Authorization` с credentials, закодированными в Base64.

**Формат:**
```
Authorization: Basic base64(логин:пароль)
```

**Пример:**
```
Authorization: Basic YWRtaW46cGFzc3dvcmQxMjM=
```

**Важно:**
- Используются логин и пароль пользователя IntraService (не API-ключ)
- Отдельный endpoint для получения токена не предусмотрен
- Права доступа определяются системной ролью пользователя и его назначениями на сервисы

## Общий вид и типы запроса.

API поддерживает стандартные HTTP методы:

*   `GET` - получение ресурсов.
*   `POST` - создание новых ресурсов.
*   `PUT` - обновление существующих ресурсов.

**Важно:**
*   Для `GET` запросов, если в URL указан `id` ресурса (например, `/api/service/1`), возвращается конкретный ресурс. Если `id` отсутствует, возвращается коллекция ресурсов.
*   Для `PUT` запросов достаточно отсылать только те поля, которые изменяются.

## Поддерживаемые форматы

*   **Получение данных (GET):** Поддерживаются форматы `json` и `xml`. Формат определяется стандартными параметрами в заголовке запроса `Accept`.
*   **Отправка данных (POST, PUT):** Используется только формат `json`. В теле запроса передается JSON-объект.

## Ответ на запрос и возврат ошибки

В ответе на запрос возвращается стандартный код состояния HTTP.

*   **Коды 2XX** - успешное выполнение запроса.
*   **Коды 4XX-5XX** - ошибка.

При успешном выполнении `POST` и `PUT` запросов в большинстве случаев возвращается созданный или измененный объект (аналогично `GET` запросу с `id`).

**Пример успешного ответа:**

```json
{
  "Id": 1,
  "Code": "SRVC",
  "Name": "Service",
  "Description": "Описание сервиса",
  "IsArchive": false,
  "IsPublic": false,
  "ParentId": 185,
  "Path": "185|1|"
}
```

**Пример ответа с ошибкой (для PUT-запроса):**

```json
{
  "Message": "The request is invalid.",
  "MessageDetail": "The parameters dictionary contains a null entry for parameter 'id' ..."
}
```

## Ограничение числа получаемых полей

Для запросов конкретных ресурсов и их коллекций можно ограничить число возвращаемых полей с помощью параметра `fields`. В нем через запятую перечисляются названия требуемых полей.

**Пример:**
```
GET /api/service?fields=Id,Name,Code
```

## Поиск и фильтрация коллекции ресурсов

Если для ресурса доступен поиск или фильтрация, это указано в его детальном описании. Параметры поиска и фильтрации передаются в строке запроса.

## Сортировка коллекций

Для сортировки коллекций используется параметр `sort`.

**Формат:**
```
?sort=FieldName1 asc, FieldName2 desc
```

Где:
- `FieldName` — название поля ресурса
- `asc` — сортировка по возрастанию
- `desc` — сортировка по убыванию

**Пример для заявок:**
```
/api/task?serviceid=31&sort=StatusId desc, PriorityId asc
```

## Особенности вывода коллекции ресурсов

Коллекции ресурсов возвращаются постранично. В ответе присутствует блок `Paginator` со следующей информацией:

| Название | Описание | Использование в запросе |
| :--- | :--- | :--- |
| `Count` | Общее число ресурсов в коллекции | |
| `Page` | Номер отображаемой страницы (по умолчанию 1) | `+` |
| `PageCount` | Количество страниц | |
| `PageSize` | Количество ресурсов на странице (по умолчанию 25, не больше 2000) | `+` |
| `CountOnPage` | Возвращенное количество ресурсов на странице | |

Размер страницы задается параметром `pagesize`, а номер страницы - параметром `page`.

**Пример запроса и ответа в JSON:**

```
GET /api/task?page=2&pagesize=10
```

```json
{
  "Tasks": [...],
  "Paginator": {
    "Count": 111786,
    "Page": 2,
    "PageCount": 11179,
    "PageSize": 10,
    "CountOnPage": 10
  }
}
```

## Мультиязычность и даты.

*   **Мультиязычные строковые данные** возвращаются в соответствии с настройкой языка у пользователя.
*   **Даты** возвращаются в соответствии с часовым поясом пользователя. При передаче дат в API они должны быть указаны в часовом поясе пользователя.

Часовой пояс может быть определен для Пользователя, Подразделения (Компании) и в Общих настройках. Приоритет имеет следующая иерархия:

1.  Настройка Пользователя.
2.  Настройка Компании (включая иерархию подразделений).
3.  Общие настройки системы.

Получить информацию о часовом поясе и языке текущего пользователя можно через специальный запрос (описан в разделе `Пользователи`).

## Формат идентификаторов

Все основные идентификаторы в API — целые числа (`int`):

```json
{
  "Id": 12345,
  "UserId": 42,
  "ServiceId": 7,
  "StatusId": 31
}
```

**Исключения:**
- `ADGuid` — идентификатор в Active Directory, передается как строка в формате `Guid`
- `FileTokens` — временные идентификаторы файлов, передаются как строка с числами через запятую (например, `"1,23"`)
- `DeviceToken` — токен мобильного устройства, передается как строка

## Rate Limiting (ограничения запросов)

В текущей версии API явные ограничения на количество запросов не документированы.

**Рекомендации:**
- Избегайте создания избыточной нагрузки (например, запросов в бесконечных циклах без пауз)
- Реализуйте механизм повторных попыток с экспоненциальной задержкой для обработки временных сбоев
- При интенсивной нагрузке используйте пагинацию для получения данных порциями

## Данные в заголовке HTTP-запросов

### Передача имени устройства и версии системы в заголовках

Для мобильных приложений в заголовках запросов можно передавать следующую информацию:

*   `Device-Name` - имя устройства (например, "iOS iPhone5").
*   `Device-Version` - версия клиентского ПО (например, "1.04").

Эта информация помогает отслеживать, с какого устройства производились изменения в заявках.

### Версия API

В заголовок каждого ответа вставляется заголовок `X-API-Version`, значение которого показывает текущую версию API.

**Пример:**
```
X-API-Version: 4.31 beta
```
