### Получение списка заявок

**Вызов:**
```
GET /api/task?serviceid={serviceid}&fields={fieldList}&search={searchString}&archive={value}&inactive={value}&sort={sortFields}&filterid={filter_id}&{filterFields}&include={includeList}&pagesize={value}&page={value}
```

**Важно:** При выводе списка заявок учитываются настройки роли на сервисе для текущего пользователя по видимости заявок.

#### Возможные параметры

Все параметры являются необязательными.

| Параметр | Описание | Пример |
| :--- | :--- | :--- |
| `serviceid` | Идентификатор сервиса для фильтрации списка заявок. Если не задан – выведет заявки всех доступных сервисов | `/api/task?serviceid=3` |
| `count` | Получить в ответе общее число заявок. Возможные значения: `True` (по умолчанию) – ограничено 1000; `False` – не высчитывает, только указывает наличие следующей страницы; `all` – подсчитывает все заявки | `/api/task?count=all` |
| `fields` | Перечень полей ресурса «Заявка» для вывода | `/api/task?fields=Id,Name,StatusId` |
| `search` | Строка поиска. Ищет в полях, указанных в таблице, а также по всем комментариям заявки | `/api/task?search=тест` |
| `archive` | Возвращать заявки архивных сервисов. `False` (по умолчанию) – только неархивные; `True` – все | `/api/task?archive=true` |
| `inactive` | Возвращать заявки неактуальных сервисов. `False` (по умолчанию) – только актуальные; `True` – все | `/api/task?inactive=true` |
| `filterid` | Идентификатор фильтра. Список заявок возвращается в соответствии с фильтром | `/api/task?filterid=45` |
| `sort` | Сортировка списка. Формат: `FieldName asc\|desc` | `/api/task?sort=StatusId desc, PriorityId asc` |
| `{filterFields}` | Поля заявки и значения для фильтрации по ним | См. раздел "Фильтрация списка заявок" |
| `pagesize`, `page` | Параметры отображения списка | `/api/task?page=2&pagesize=10` |
| `include` | Позволяет получить связанные ресурсы в одном запросе. Возможные значения: `ASSETIDS`, `CATEGORYIDS`, `PRIORITY`, `SERVICE`, `STATUS`, `USER` | `/api/task?include=status,priority` |

#### Параметр `include`

Позволяет включить в ответ связанные ресурсы:

| Значение | Описание |
| :--- | :--- |
| `ASSETIDS` | Выводит поле `AssetIds` (идентификаторы активов) для каждой заявки |
| `CATEGORYIDS` | Выводит поле `CategoryIds` (идентификаторы категорий) для каждой заявки |
| `PRIORITY` | Включает блок `<Priorities>` с подробной информацией о приоритетах |
| `SERVICE` | Включает блок `<Services>` с подробной информацией о сервисах |
| `STATUS` | Включает блок `<Statuses>` с подробной информацией о статусах |
| `USER` | Включает блок `<Users>` с информацией о заявителях, исполнителях, наблюдателях и согласующих |

**Пример ответа с `include=status,priority`:**

```json
{
  "Tasks": [
    { "StatusId": 7, "PriorityId": 12, ... },
    { "StatusId": 8, "PriorityId": 12, ... }
  ],
  "Statuses": [
    { "Id": 7, "Name": "Открыта", ... },
    { "Id": 8, "Name": "В работе", ... }
  ],
  "Priorities": [
    { "Id": 12, "Name": "Высокий", ... }
  ],
  "Paginator": { ... }
}
```

**Пример в XML:**

```xml
GET /api/task?fields=Id,Name,ServiceId&include=service&pagesize=1&page=2

<TaskList>
  <Tasks>
    <Task>
      <Id>158</Id>
      <Name>test category</Name>
      <ServiceId>16</ServiceId>
    </Task>
  </Tasks>
  <Services>
    <Service>
      <Id>16</Id>
      <Code>c2</Code>
      <Name>сервис 2</Name>
      <Description/>
      <IsArchive>False</IsArchive>
      <IsPublic>False</IsPublic>
      <ParentId/>
      <Path>16</Path>
    </Service>
  </Services>
  <Paginator>
    <Count>11</Count>
    <Page>2</Page>
    <PageCount>11</PageCount>
    <PageSize>1</PageSize>
    <CountOnPage>1</CountOnPage>
  </Paginator>
</TaskList>
```

| `Changed` | DateTime | Дата последнего изменения заявок равна указанному дню |
| `CreatedLessThan` | DateTime | Дата создания заявок меньше или равна указанной |

| `Closed` | DateTime | Дата закрытия заявок равна указанному дню |
| `DeadlineLessThan` | DateTime | Плановый срок выполнения заявок меньше или равен указанному |
| `DeadlineMoreThan` | DateTime | Плановый срок выполнения заявок больше или равен указанному |
| `Deadline` | DateTime | Плановый срок выполнения заявок равен указанному дню |
| `ReactionDateLessThan` | DateTime | Плановый срок реакции заявок меньше или равен указанному |
| `ReactionDateMoreThan` | DateTime | Плановый срок реакции заявок больше или равен указанному |
| `ReactionDate` | DateTime | Плановый срок реакции заявок равен указанному дню |
| `ReactionDateFactLessThan` | DateTime | Фактический срок реакции заявок меньше или равен указанному |
| `ReactionDateFactMoreThan` | DateTime | Фактический срок реакции заявок больше или равен указанному |
| `ReactionDateFact` | DateTime | Фактический срок реакции заявок равен указанному дню |

| `ResolutionDateFact` | DateTime | Фактический срок выполнения заявок равен указанному дню |
| `ResolutionOverdue` | Bool | Заявки, срок выполнения которых истек |
| `ReactionOverdue` | Bool | Заявки, срок реакции которых истек |
| `TypeIds` | String | Заявки в указанных типах. Перечисление идентификаторов через запятую |

| `PriorityIds` | String | Заявки с указанными приоритетами. Перечисление идентификаторов через запятую |
| `ServiceIds` | String | Заявки с указанными сервисами. Перечисление идентификаторов через запятую |
| `EditorIds` | String | Заявки, у которых последний изменивший пользователь входит в указанный набор |

| `ExecutorIds` | String | Заявки, на которые назначен хотя бы один исполнитель из заданного перечня |
| `ExecutorGroupIds` | String | Заявки, на которые назначена одна из групп исполнителей из заданного перечня |
| `ObserverIds` | String | Заявки, на которые назначен хотя бы один наблюдатель из заданного перечня |

| `AssetIds` | String | Заявки, на которые назначен хотя бы один актив из заданного перечня |

#### Примеры запросов с фильтрацией

**Пример 1:** Заявки, изменявшиеся после 13 ноября 2015 года 10:00 со статусами 1 или 7:

```
GET /api/task?ChangedMoreThan=2015-11-13 10:00&StatusIds=1,2
```

**Пример 2:** Заявки с категориями 18, 22, 10 и исполнителем 215:

```
GET /api/task?CategoryIds=18,22,10&ExecutorIds=215
```

### Получение конкретной заявки

**Вызов:**
```
GET /api/task/{taskid}?include={includeList}
```

Где `{taskid}` – идентификатор (номер) заявки.

#### Параметр `include`

Возможные значения:

| Значение | Описание |
| :--- | :--- |
| `PRIORITY` | Включает блок `<Priorities>` с информацией о приоритете заявки |
| `SERVICE` | Включает блок `<Services>` с информацией о сервисе заявки |
| `STATUS` | Включает блок `<Statuses>` с информацией о статусе заявки |
| `USER` | Включает блок `<Users>` с информацией о заявителе, исполнителях, наблюдателях и согласующих |
| `TASKTYPE` | Включает блок `<TaskType>` с информацией о типе заявки (включая дополнительные поля) |
| `TASKTYPESETTINGS` | Включает блок `<TaskTypeSettings>` со значениями бизнес-настроек из типа заявки |
| `USERTASKRIGHTS` | Включает блок `<UserTaskRights>` с полномочиями пользователя по работе с заявкой |

**Пример ответа в XML:**

```xml
GET /api/task/159?include=status,priority

    <Closed/>
    <CompletionStatus/>
    <CoordinatorIds/>
    <Coordinators/>
    <Created>26.11.2015 16:18:06</Created>
    <Creator>Администратор</Creator>
    <CreatorADGuid/>
    <CreatorComments/>
    <!-- ... остальные поля ... -->
  </Task>
  <Statuses>
    <Status>
      <Id>31</Id>
      <Name>Открыта</Name>
      <!-- ... -->
    </Status>
  </Statuses>
  <Priorities>
    <Priority>
      <Id>9</Id>
      <Name>Средний</Name>
      <!-- ... -->
    </Priority>
  </Priorities>
</TaskForm>
```

### Получение шаблона заявки для создания (значения по умолчанию)

**Вызов:**
```
GET /api/newtask?serviceid={serviceid}&tasktypeid={tasktypeid}&include={includeList}
```

#### Параметры

| Параметр | Обязательность | Описание |
| :--- | :---: | :--- |
| `serviceid` | Да | Идентификатор сервиса, в котором создается заявка |
| `tasktypeid` | Да | Идентификатор типа заявки |
| `include` | Нет | Аналогично, как и для получения конкретной заявки |

**Вызов:**
```

```

В теле запроса передаются поля объекта Task (сам объект Task передавать не надо). Допускается передавать только те поля, которые содержат значения.

### Создание заявки от лица, которое идентифицируется электронным адресом

**Вызов:**
```
POST /api/task
```

Для создания заявки от лица пользователя, который идентифицируется электронным адресом, необходимо в теле запроса передать дополнительные поля:

| Название | Тип данных | Обязательность | Описание |
| :--- | :--- | :---: | :--- |
| `UserEmail` | string | + | Электронный адрес пользователя |
| `UserPassword` | string | + | Пароль пользователя (если создаем от пользователя, которого нет в системе) |
| `UserConfirmPassword` | string | + | Подтверждение пароля |
| `UserComments` | String | | Комментарий в карточке пользователя |
| `UserCompanyId` | int | + | Компания нового пользователя |
| `UserCurrentLanguage` | String | | Язык нового пользователя |
| `UserLogin` | String | | Логин нового пользователя (если пусто, возьмется UserEmail) |
| `UserMobilePhone` | String | | Мобильный телефон нового пользователя |
| `UserName` | String | | Имя нового пользователя (если пусто, возьмется UserEmail) |
| `UserPhone` | String | | Телефон нового пользователя |
| `UserPosition` | String | | Должность нового пользователя |
| `UserRoleId` | int | + | Системная роль нового пользователя |
| `UserTimeZone` | string | | Часовой пояс нового пользователя |

**Алгоритм создания заявки:**

1. Если в POST запросе передается электронный адрес `UserEmail`, проверяем наличие пользователя в системе с таким адресом.
2. Если пользователь **есть** – заявка создается от лица этого пользователя.
3. Если пользователя **нет** – система пытается создать пользователя, а затем заявку от его имени. В этом случае обязательно нужно передать `UserPassword` и `UserConfirmPassword`.

**Примечание:** Для упрощенной схемы создания заявки от другого лица используйте передачу поля `CreatorId`. В этом случае учитываются права текущего пользователя на изменение заявителя.

```
PUT /api/task/{taskid}
```

Где `{taskid}` – идентификатор изменяемой заявки.

### Отслеживание изменений другим пользователем

Для отслеживания изменений другим пользователем в момент сохранения нужно передавать поле `Changed`, которое мы получаем в момент получения заявки.

**Вариант 1:** Передаем поле `Changed`. Если другой пользователь изменил заявку, возникает ошибка:

```
NetworkError: 409 Conflict - /api/task/140134
```

```json
{
  "Message": "Пользователь admin изменил заявку 26.12.2013 8:18, пока вы ее редактировали."
}
```

**Вариант 2:** Не передаем поле `Changed`. Тогда изменения предыдущего пользователя будут перезаписаны.

## Прикрепление файла при создании или изменении заявки

Прикрепление файлов происходит в 2 этапа:

### Этап 1: Сохранение файлов и получение уникальных идентификаторов

**Вызов:**
```
POST /api/TaskFile
```

**Параметры запроса:**
1. Файлы (количество не ограничено)
2. Требуемое имя (необязательный параметр)

**Имя формируется следующим образом:**
- Файлы заявки: `{filekey}Name`
- Файлы в дополнительных полях: `{filekey}TaskTypeFieldId`

Имя файла должно быть без расширения.

**Пример формы:**
```html
<input name="file0" type="file"/>
<input name="file0Name" type="text"/>

<input name="file1" type="file"/>
<input name="file2" type="file"/>
<input name="file2Name" type="text"/>

<input name="file2TaskTypeFieldId" type="text" value="15"/>
```

**Ответ:**
```json
{"FileTokens":"1,23"}
```

**Важно:** В заголовках запроса необходимо указывать `content-type: multipart/form-data`.

### Этап 2: Прикрепление сохраненных файлов к заявке

Полученные ранее идентификаторы файлов `FileTokens` передаются в запросе для создания или изменения заявки в поле `FileTokens`.

## Согласование

### Перевод или создание в статусе с признаком «Согласование»

При переводе заявки в статус с признаком «Согласование» или создании заявки в нем возможны 2 варианта:

**Вариант 1:** Указаны `CoordinatorIds`, и:
- хотя бы один из указанных пользователей принадлежит сервису
- текущий пользователь может изменять список согласующих
- согласование является простым (не несколько согласований друг за другом)

Если условия выполняются, заявка перейдет в согласование с указанными согласующими.

**Вариант 2:** Во всех остальных случаях (не указаны CoordinatorIds, не привязаны к сервису, нет прав, несколько согласований) — для каждого согласования определяется список согласующих по умолчанию (по настройкам перехода в Бизнес-процессе).

**Важные замечания:**
1. Если согласований несколько подряд – переданный список `CoordinatorIds` игнорируется.
2. Если хотя бы для одного согласования не найден список согласующих по умолчанию – будет ошибка: «для согласования обязательно определение согласующих».

### Согласование заявки

**Вызов:**
```
PUT /api/task/{taskid}
```

Где `{taskid}` – идентификатор заявки в статусе «Согласование».

**Примеры:**

1. Согласуем заявку № 143:
```json
PUT /api/task/143
{"Coordinate": true}
```

2. Откажем в согласовании заявки № 145 с комментарием:
```json
PUT /api/task/145
{"Coordinate": false, "Comment": "Причина отказа от согласования: документы составлены неверно"}
```

### Прогресс согласования заявки

При вызове информации по конкретной заявке, если она находится в статусе согласования, в ответе присутствуют:

| Поле | Описание |
| :--- | :--- |
| `Coordinators` | Имена согласующих через запятую |
| `CoordinatorIds` | Идентификаторы согласующих через запятую |
| `IsCoordinatedForCoordinators` | Признаки согласования (True/False) через запятую. Порядок соответствует `Coordinators` и `CoordinatorIds` |
| `IsCoordinatedByCurrentUser` | Согласована ли заявка текущим пользователем |

**Пример:**
```xml
<CoordinatorIds>5304, 5094, 5224</CoordinatorIds>
<Coordinators>Иван Никифоров, Петр Иванов, Вячеслав Смирнов</Coordinators>
<IsCoordinatedForCoordinators>False, True, False</IsCoordinatedForCoordinators>
```

Означает:
- Иван Никифоров (ID 5304) — не согласовал
- Петр Иванов (ID 5094) — согласовал
- Вячеслав Смирнов (ID 5224) — не согласовал

## Получение связанных ресурсов по заявке


**Параметры:** Аналогично, как и для получения списка заявителей.
